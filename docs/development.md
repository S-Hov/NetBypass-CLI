# Разработка и сборка

## Стек

- C# и .NET 10;
- WPF;
- простая MVVM-структура;
- xUnit.

## Структура решения

```text
NetBypass.sln
├── src/NetBypass.App       WPF-интерфейс и сервисные модули
├── src/NetBypass.Core      hosts, профили, DoH и сетевая диагностика
└── tests/NetBypass.Tests   модульные тесты
```

## Архитектурное направление

NetBypass развивается не как собственный Anti-DPI-движок, а как управляющий
центр для нескольких стратегий:

- встроенный `hosts`-режим;
- DoH/локальный DNS;
- релей для сервисов, которым нужен другой регион;
- внешние движки GoodbyeDPI, ByeDPI и zapret через адаптеры.

Исходники внешних движков не должны копироваться в основной код без отдельного
решения. Предпочтительный путь — адаптер, который скачивает или использует
проверенный бинарник, проверяет его контрольную сумму, запускает с нужным
профилем и полностью очищает созданные ресурсы после отключения.

## Команды

```powershell
dotnet restore
dotnet build
dotnet test
dotnet run --project .\src\NetBypass.App
```

Для запуска интерфейса требуется Windows. Тесты Core используют временные
файлы и не изменяют системный `hosts`.

## Основные правила

- изменять только блок `NETBYPASS-BEGIN / NETBYPASS-END`;
- сохранять строки пользователя вне блока;
- создавать резервную копию до первой записи;
- проверять записанный результат;
- не доверять сохранённому флагу состояния;
- не применять изменение чекбоксов до явного сохранения;
- отклонять локальные и некорректные назначения модулей.
- не применять статический IP, не прошедший TCP/TLS-проверку.
- перед внешними Anti-DPI-движками сначала реализовать журнал ресурсов;
- все процессы, службы, драйверы, DNS-настройки и временные файлы должны
  удаляться через проверяемый cleanup-сценарий.

Диагностика зависит от интерфейсов `IDohResolver` и `IEndpointProbe`, поэтому
модульные тесты не должны обращаться к реальной сети.

## Проверка изменений

Перед коммитом:

```powershell
dotnet build
dotnet test
git diff --check
```

При изменении логики `hosts` добавляйте тесты как минимум для:

- пустого состояния;
- активного блока;
- отличающегося выбора;
- повреждённых маркеров;
- сохранения пользовательских строк;
- восстановления после повреждения.
