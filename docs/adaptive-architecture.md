# Архитектура и план развития NetBypass

## 1. Новая продуктовая идея

NetBypass не должен становиться ещё одним собственным Anti-DPI-движком.
Главная ценность проекта — сделать удобный управляющий центр для уже
существующих инструментов и скрыть от пользователя сложность командной строки.

Пользователь выбирает не технический метод, а нужные сервисы:

```text
ChatGPT  ✓
Discord  ✓
YouTube  ✓
```

NetBypass сам выполняет диагностику, определяет тип проблемы и выбирает самый
простой подходящий способ:

```text
Проблема с DNS       → DoH или локальный DNS
Нужен другой регион  → релей
Блокировка через DPI → GoodbyeDPI, ByeDPI или zapret
```

В интерфейсе пользователь должен видеть короткий понятный результат:

```text
Защита включена
Движок: zapret
Профиль: Discord
Проверка пройдена
```

NetBypass остаётся не VPN:

- он не обязан перенаправлять весь трафик компьютера;
- пользователь выбирает конкретные сервисы;
- программа показывает применённую стратегию;
- все изменения должны отключаться и проверяться после отключения.

## 2. Почему это направление сильнее

zapret, GoodbyeDPI и похожие проекты уже решают сложную Anti-DPI-часть.
Их главная проблема для обычного пользователя — настройка:

- много параметров запуска;
- разные профили под разные сервисы;
- отдельные команды для установки и удаления служб;
- работа с WinDivert и системными сетевыми фильтрами;
- необходимость понимать, чем DNS-проблема отличается от DPI-проблемы.

NetBypass должен закрыть именно этот слой:

- выбрать профиль;
- скачать или найти проверенный движок;
- сформировать параметры запуска;
- запустить движок;
- проверить, что выбранный сервис реально заработал;
- остановить движок;
- удалить созданные службы, процессы, драйверы и правила;
- восстановиться после аварийного закрытия.

Итоговая специализация:

```text
NetBypass = UI + диагностика + выбор стратегии + безопасное управление движками
```

А не:

```text
NetBypass = собственный Anti-DPI с нуля
```

## 3. Целевая архитектура

```text
NetBypass
├── WPF UI
│   ├── Главная
│   ├── Сервисы
│   ├── Диагностика
│   └── Журнал очистки
│
├── Core Engine
│   ├── Service Profiles
│   ├── Diagnostics
│   ├── Strategy Detector
│   ├── Connection Plan
│   └── Resource Journal
│
├── Built-in Strategies
│   ├── Hosts adapter
│   ├── DoH / DNS adapter
│   └── Relay adapter
│
└── External Engines
    ├── GoodbyeDPI adapter
    ├── ByeDPI adapter
    └── zapret adapter
```

### WPF UI

Интерфейс отвечает только за управление и отображение:

- какие сервисы выбраны;
- какая стратегия применена;
- какой движок запущен;
- какой профиль используется;
- прошла ли проверка;
- что было создано в системе;
- всё ли удалено после отключения.

UI не должен напрямую писать `hosts`, запускать внешние процессы или менять DNS.
Он обращается к Core Engine.

### Core Engine

Центральный управляющий слой:

- загружает профили сервисов;
- запускает диагностику;
- определяет тип проблемы;
- строит `ConnectionPlan`;
- выбирает встроенную стратегию или внешний движок;
- применяет план;
- проверяет результат после применения;
- записывает все созданные ресурсы в журнал;
- выполняет безопасное отключение и восстановление.

### Built-in Strategies

Встроенные стратегии — это то, что NetBypass умеет делать сам:

- временно применять записи в `hosts`;
- проверять адреса через DoH, TCP, TLS и HTTP;
- выбирать рабочий релей из списка;
- очищать DNS-кеш;
- в будущем — управлять локальным DNS, если это будет нужно.

### External Engines

Внешние движки не встраиваются исходниками в NetBypass. Они подключаются через
адаптеры.

Адаптер отвечает за жизненный цикл конкретного инструмента:

- загрузить проверенную версию;
- проверить контрольную сумму или подпись;
- сохранить тексты лицензий и notices;
- сформировать параметры запуска;
- запустить процесс или службу;
- проверить статус;
- собрать лог;
- остановить;
- удалить службу, WinDivert и временные файлы;
- доказать, что после отключения ничего не осталось.

## 4. Диагностика и выбор стратегии

NetBypass не должен включать тяжёлый Anti-DPI-движок сразу. Сначала программа
должна понять, что именно сломано.

Базовый алгоритм:

```text
1. Проверить выбранные сервисы.
2. Получить DNS-ответы через DoH.
3. Проверить TCP-подключение.
4. Проверить TLS-сертификат с правильным именем домена.
5. Проверить HTTP-ответ, если это безопасно для сервиса.
6. Сравнить результат прямого подключения и релея.
7. Если проблема похожа на DPI — предложить или включить внешний движок.
8. После применения стратегии снова проверить сервис.
```

Приоритет стратегий:

```text
Direct / DoH → Hosts → Relay → GoodbyeDPI / ByeDPI → zapret
```

Общий принцип: сначала использовать самый простой и обратимый способ, а более
глубокие сетевые вмешательства включать только тогда, когда они действительно
нужны.

## 5. ConnectionPlan

Программа должна хранить не просто флаг `включено`, а понятный план подключения.

Пример:

```json
{
  "service": "discord",
  "strategy": "ExternalEngine",
  "engine": "zapret",
  "profile": "discord-default",
  "reason": "TLS доступен, но HTTP/2 соединение сбрасывается после ClientHello",
  "verifiedAt": "2026-06-24T18:00:00+03:00",
  "expiresAt": "2026-06-25T18:00:00+03:00"
}
```

План нужен, чтобы:

- объяснить пользователю, что включено;
- повторно проверить результат после перезапуска;
- понять, что нужно отключать;
- не применять устаревшую или непроверенную конфигурацию.

## 6. Resource Journal

Это самая важная часть перед интеграцией Anti-DPI.

Anti-DPI-движки работают глубже, чем текущий `hosts`. После них могут остаться:

- запущенный процесс;
- Windows-служба;
- загруженный WinDivert-драйвер;
- временные файлы;
- правила фильтрации;
- изменённый DNS;
- изменённый `hosts`;
- конфликт с VPN, антивирусом или другим Anti-DPI.

Поэтому NetBypass должен вести журнал созданных ресурсов:

```text
Запущен процесс 1428
Установлена служба NetBypassZapret
Загружен WinDivert
Создан профиль Discord
Изменён hosts
Очищен DNS-кеш
```

При выключении программа должна пройти по журналу в обратном порядке:

```text
Остановить процесс
Удалить службу
Выгрузить WinDivert
Удалить временные файлы
Восстановить DNS
Удалить блок NetBypass из hosts
Проверить, что ничего не осталось
```

После аварийного закрытия следующий запуск должен начинаться с восстановления:

```text
Обнаружены незавершённые изменения NetBypass
Проверить и очистить
```

Критерий готовности: после отключения NetBypass не оставляет процессов, служб,
драйверов, DNS-настроек, временных файлов и записей в `hosts`, кроме явно
сохранённых пользовательских настроек.

## 7. Интерфейс внешнего движка

Перед интеграцией конкретных инструментов нужен общий контракт.

Минимальная модель:

```csharp
public interface IExternalBypassEngine
{
    string Id { get; }
    string DisplayName { get; }

    Task<EngineAvailability> CheckAvailabilityAsync(CancellationToken cancellationToken);
    Task<EngineInstallResult> InstallAsync(CancellationToken cancellationToken);
    Task<EngineCompatibility> CheckCompatibilityAsync(CancellationToken cancellationToken);
    Task<EngineRunResult> StartAsync(EngineProfile profile, CancellationToken cancellationToken);
    Task<EngineStatus> GetStatusAsync(CancellationToken cancellationToken);
    Task<EngineStopResult> StopAsync(CancellationToken cancellationToken);
    Task<EngineCleanupResult> CleanupAsync(CancellationToken cancellationToken);
    Task<IReadOnlyList<EngineLogEntry>> GetLogsAsync(CancellationToken cancellationToken);
}
```

Адаптеры не должны протекать в UI. Для интерфейса все движки выглядят одинаково:

```text
Движок: GoodbyeDPI
Статус: работает
Профиль: Discord
Последняя проверка: успешно
```

## 8. Порядок интеграции движков

### GoodbyeDPI / ByeDPI первым

GoodbyeDPI и ByeDPI подходят как первый внешний движок, потому что они проще для
Windows-сценария:

- их легче запускать как отдельный процесс;
- проще спрятать параметры за профилями;
- проще проверить старт, стоп и очистку;
- меньше архитектурных слоёв, чем у zapret.

Цель первой интеграции — не покрыть все возможные режимы, а доказать, что
NetBypass умеет безопасно управлять внешним Anti-DPI-движком.

### zapret вторым

zapret мощнее и гибче, поэтому его стоит добавлять после того, как уже есть:

- общий интерфейс внешнего движка;
- журнал ресурсов;
- проверка совместимости;
- безопасное выключение;
- UI для профилей;
- тестовый сценарий очистки.

zapret должен подключаться как продвинутый движок, а не как набор случайных
команд в коде приложения.

## 9. Лицензии и поставка бинарников

Исходники внешних проектов не следует копировать внутрь NetBypass без крайней
необходимости. Предпочтительный вариант — скачивать или поставлять проверенные
бинарники и управлять ими как отдельными компонентами.

На момент актуализации документации:

- zapret указывает лицензию MIT;
- GoodbyeDPI указывает лицензию Apache-2.0.

Это позволяет интеграцию, но накладывает обязательства:

- сохранять тексты лицензий;
- сохранять сведения об авторах;
- сохранять notices, если они есть;
- отдельно проверять лицензии зависимостей, включая WinDivert;
- показывать пользователю, какой внешний компонент используется.

Перед релизом с внешними бинарниками нужен отдельный аудит зависимостей.

## 10. Новый план реализации

### Этап 1. Стабилизировать текущую базу

- довести UI состояний: выключено, проверка, активно, частично активно, ошибка;
- гарантировать корректную очистку `hosts`;
- очищать DNS-кеш после включения и выключения;
- показывать пользователю понятные причины ошибок;
- покрыть отключение и восстановление тестами.

Критерий готовности: при выключении программа возвращается в состояние
`Не настроено`, не выглядит как ошибка и не оставляет управляемых записей.

### Этап 2. Автовыбор релея

- хранить несколько кандидатов релея для сервиса;
- проверять их параллельно через TCP/TLS/HTTP;
- выбирать рабочий адрес;
- кешировать последний рабочий вариант;
- объяснять причину выбора.

Критерий готовности: если основной релей недоступен, NetBypass выбирает другой
рабочий адрес при следующей диагностике.

### Этап 3. Strategy Detector и ConnectionPlan

- определять тип проблемы: DNS, регион, DPI, неизвестная причина;
- строить `ConnectionPlan`;
- хранить срок действия плана;
- после применения повторно проверять сервис;
- показывать стратегию в UI.

Критерий готовности: пользователь видит не только `включено`, но и `как именно
работает подключение`.

### Этап 4. Resource Journal и Safe Cleanup

- записывать все созданные ресурсы;
- отключать их в обратном порядке;
- проверять отсутствие хвостов;
- восстанавливаться после аварийного завершения;
- добавить страницу или блок `Журнал очистки`.

Критерий готовности: программа умеет доказать, что после выключения ничего не
оставила.

### Этап 5. Engine Adapter API

- добавить общий интерфейс внешних движков;
- добавить модель профиля движка;
- добавить проверку совместимости;
- добавить сбор логов;
- отделить установку, запуск, остановку и очистку.

Критерий готовности: можно подключить тестовый fake-движок и полностью пройти
цикл install → start → verify → stop → cleanup без реального Anti-DPI.

### Этап 6. GoodbyeDPI / ByeDPI adapter

- загрузка или обнаружение бинарника;
- проверка контрольной суммы;
- хранение лицензии;
- запуск с безопасным профилем;
- проверка выбранного сервиса после запуска;
- остановка и очистка.

Критерий готовности: пользователь может включить профиль для одного сервиса и
выключить его без ручной работы в командной строке.

### Этап 7. zapret adapter

- поддержать Windows-сборку zapret;
- описать профили для Discord/YouTube и других сервисов;
- управлять службой или процессом через адаптер;
- проверять и удалять WinDivert;
- показывать диагностический лог понятным языком.

Критерий готовности: zapret запускается, проверяется и удаляется через единый
механизм NetBypass.

### Этап 8. Автоматический подбор профиля

- запускать диагностику перед выбором движка;
- пробовать безопасные профили по очереди;
- сохранять последний рабочий профиль;
- не применять заведомо нерабочий вариант;
- объяснять пользователю, почему выбран именно этот профиль.

Критерий готовности: пользователь выбирает сервис, а NetBypass сам подбирает
подходящий профиль без ручного чтения документации zapret или GoodbyeDPI.

## 11. Текущее состояние проекта

Сейчас реализован ранний базовый слой:

- WPF-интерфейс с главной страницей, сервисами и диагностикой;
- загрузка `.hosts`-модулей;
- запись и удаление управляемого блока `hosts`;
- проверка DoH, TCP, TLS и HTTP;
- сохранение результатов диагностики;
- очистка DNS-кеша Windows;
- базовые тесты для `hosts`, модулей и диагностики.

Проект пока не умеет:

- выбирать релей из нескольких кандидатов;
- строить полноценный `ConnectionPlan`;
- определять тип ограничения;
- управлять внешними Anti-DPI-движками;
- вести журнал всех созданных системных ресурсов;
- доказывать очистку процессов, служб и драйверов после отключения.

Следующий практический шаг: закрыть этап 1 и начать этап 2 — стабилизировать
UI/очистку текущего hosts-режима, затем добавить модель кандидатов релея и
автоматический выбор рабочего адреса.
