# Диагностика подключения

NetBypass не считает наличие записей в `hosts` доказательством
работоспособности.

## Что проверяется

Для каждого выбранного сервиса создаётся `ServiceProfile` версии 1. Текущие
`.hosts`-модули остаются источником доменов и статических адресов.

Для каждого уникального IP внутри профиля выполняются:

1. DoH-запрос к Cloudflare и Google для получения настоящих адресов домена;
2. TCP-подключение к статическому адресу модуля;
3. TLS handshake с настоящим именем домена в SNI;
4. проверка сертификата;
5. HTTP `HEAD` для получения дополнительного статуса.

Адрес считается пригодным для текущего `hosts`-режима только после успешных
TCP и TLS. HTTP используется как дополнительная информация: например,
Cloudflare может вернуть `403`, хотя сеть и TLS работают.

## Поведение при включении

Перед записью выбранных модулей NetBypass проверяет все их уникальные адреса.
Если часть профилей не прошла проверку:

- записи применяются только для доступных сервисов;
- на главной показывается количество доступных сервисов от общего выбора;
- отдельная кнопка открывает список успешных и неуспешных результатов;
- недоступные профили не попадают в управляемый блок `hosts`.

Если недоступны все выбранные профили, новая конфигурация не записывается.

Во время проверки интерфейс показывает вращающийся индикатор, общий прогресс и
добавляет результаты по мере завершения сервисов.

Если часть сервисов недоступна, действие **Исключить недоступные и применить**
снимает их выбор, повторно проверяет оставшиеся профили и применяет только
рабочую конфигурацию. Повторная диагностика сама по себе не может исправить
неработающий статический IP — для этого потребуется обновление модуля или
будущий механизм Adaptive Hosts.

Результаты сохраняются в:

```text
%LOCALAPPDATA%\NetBypass\diagnostics.json
```

Проверка считается актуальной шесть часов. После этого интерфейс показывает,
что записи применены, но проверка устарела.

## Ограничения первого этапа

- статические адреса пока не заменяются автоматически;
- альтернативный SNI-релей не выбирается;
- DoH используется для сравнения и диагностики, а не для нового маршрута;
- активный VPN или TUN влияет на результат проверки;
- успешные TCP и TLS не гарантируют отсутствие регионального ограничения
  внутри сайта.

Автоматический выбор адреса и релея относится к этапу **Adaptive Hosts**.
